Developer Note — "Send Verification Code" action
  1. Validate Solax account: Before sending anything, the email must be checked against the Solax Cloud API GET /account/exists?email={email} to confirm it is a registered user account. If no match is found, surface an error: "We couldn't find a Solax account with this email. Please check and try again."
  2. Send consent email: If the account is confirmed, trigger a transactional email (e.g. SendGrid) to the customer. The email body must contain a full legal consent disclaimer (to be drafted by legal team), explaining precisely what data Reposit will access, how it will be used, and the rights of the customer. At the bottom of the email, the consent code (a time-limited one-time token, e.g. 6-digit TOTP or UUID prefix) is displayed. The email must make clear: "By entering this code on the Reposit website, you grant consent to the terms in this document."
  3. Code expiry: The consent code should expire after 15 minutes. Store the code server-side (e.g. Redis with TTL) keyed to the email. Do not expose the code in the front-end URL or JS.

Developer Note — "Authorise & Analyse" action
  1. Verify code server-side: Cross-check the submitted code against the stored token for this email. If invalid or expired, return a clear error. Do not process anything further.
  2. Fire consent hard copy: On successful verification, immediately generate and distribute a PDF consent agreement (timestamped, with IP, email, and code confirmation) via email to three parties: the customer, Reposit, and the Solax OEM representative. Use a document-generation service (e.g. DocuSeal, WeasyPrint microservice). Store a copy in the customer record.
  3. Enqueue Solax data fetch: Do not perform the Solax API call synchronously from the frontend. Instead, enqueue a background job using Celery (with Redis broker) to call the Solax Cloud API GET /cloud/v1/datapoint/list for the past 12 months of interval data. This ensures persistence and volume management — if the user closes the browser, the job continues. On job completion, trigger the next pipeline stage via webhook or polling.
  4. Run the Optimiser: Once data is received, pass it to the Reposit optimiser with appropriate constraints set for this customer profile (usage shape, peak/off-peak ratio, battery capacity). The optimiser returns a set of feasible offer parameters.
  5. Run the Product Utility Calculator: The optimiser output is fed into the conjoint survey-derived utility calculator to rank offer combinations and select the top 3 by estimated customer utility (WTP). These are the offers shown on the next page.
  6. Email fallback: If the user closes the browser before reaching the offers page, the system should always email the calculated offers when ready. Consider making this the default for all users — send the offer email regardless, and deep-link back into the journey at Step 5.

Developer Note — Offer Generation Logic
  1. Wholesale Simulation: Run a digital twin simulation in the backend to estimate:
    • Wholesale Revenue: Estimated revenue generated from battery exports.
    • Wholesale Cost: Estimated cost associated with energy imports.
  2. Calculate Wholesale Margin: Calculate Wholesale Margin = Wholesale Revenue - Wholesale Cost. This margin must be shown to the user in both real terms ($) and as a percentage (%).
  3. Apply Internal Margin Constraint: Define our required internal margin percentage.
  4. Calculate Monthly Fee Requirement: The necessary Monthly Fee is calculated as the difference between the Estimated Wholesale Margin and our Required Internal Margin. This sets a hard constraint for the package generation model.
  5. Product Utility Optimization: We use an additive utility model based on conjoint survey data to evaluate available combinations. Use the exact maths below:
    • Part-Worth Utilities: Extract the utility values (ranging from 0.0 to 100.0) from the conjoint dataset for each attribute (Installation Cost, Monthly Fee, Signup Incentive, FAIR Use Cap, Contract Term).
    • Total Utility Calculation: Total Utility = U(Install_Cost) + U(Monthly_Fee) + U(Sign_Incentive) + U(FAIR_Cap) + U(Contract_Term).
    • Average Utility (Score): Score = Total Utility / Number of Attributes. Compute the final percentage score (0-100%) for each generated package.
    • Optimization Logic: Iterate over all possible combinations of the product attributes, filtering out any permutations that don't satisfy the "Monthly Fee Requirement" constraint derived from the Digital Twin simulation. Compute the Score for all valid permutations and return the top 3 scoring packages.
